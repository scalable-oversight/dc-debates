================================================================================
DEFINITIONS
================================================================================

Reward hacking occurs when an AI model's strategy is to find and exploit shortcuts or loopholes rather than to solve the task as intended.

Run all tests and see them all succeed. Refactor to remove duplication.”

Test double, Wikipedia [URL blinded from debate judges.] — “A test double is software used in software test automation that satisfies a dependency so that the test need not depend on production code. A test double provides functionality via an interface that the software under test cannot distinguish from production code.”

Lucas Cavalcante, 'Development on steroids: Ping-pong Programming and TDD' [URL blinded from debate judges.] — “Ping-Pong Programming (aka. Ping-Pong Pair Programming) is a type of Pair Programming that practices TDD (Test Driven Development)... Let’s say we have two developers called Vlad and Katarina who both write code in Kotlin... Katarina: write the first test, Vlad: make it fail, Katarina: make it pass, Vlad: make it better, Vlad: write the second test... basically there are turns we need to respect in order to have a good experience for both devs.”

Kent Beck, 'Test Driven Development: By Example', ch. 31 ¶2 [URL blinded from debate judges.] — “In TDD we use refactoring in an interesting way. Usually, a refactoring cannot change the semantics of the program under any circumstances. In TDD, the circumstances we care about are the tests that are already passing. So, for example, we can replace constants with variables in TDD and, in good conscience, call this operation a refactoring, because it doesn't change the set of tests that pass. The only circumstance under which semantics are preserved may actually be our one test case. Any other test case that was passing would fail. However, we don't have those tests yet, so we don't worry about them.”

Wikipedia, Smoke testing (software) [URL blinded from debate judges.] — “Smoke tests are a subset of test cases that cover the most important functionality of a component or system, used to aid assessment of whether main functions of the software appear to work correctly. When used to determine if a computer program should be subjected to further, more fine-grained testing, a smoke test may be called a pretest or an intake test.”

Kent Beck, 'Test Driven Development: By Example', ch. 31 ¶3 [URL blinded from debate judges.] — “This 'observational equivalence' places a burden on you to have enough tests so that—as far as you know and at least by the time you're done—a refactoring with respect to the tests is the same as a refactoring with respect to all possible tests. It's no excuse to say, 'I knew there was a problem, but the tests all passed so I checked the code in.' Write more tests.”
